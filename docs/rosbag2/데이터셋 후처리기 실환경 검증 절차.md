# 데이터셋 후처리기 실환경 검증 절차

본 문서는 [데이터셋 후처리기 설계서](./데이터셋%20후처리기%20설계서.md) 의 Phase 1~3 구현을 **실 인프라(Docker PostgreSQL · 실 rosbag · 실 성능 측정)** 로 검증하기 위한 절차를 정리한 **실행용 runbook** 이다.

개발 환경의 mock 기반 단위/통합 테스트 (`pytest rdfp/postproc/tests/`) 는 완료되어 있으며, 본 문서는 그 다음 단계에서 **실 운영 환경이 준비되었을 때 즉시 수행 가능한** 검증 작업만 다룬다.

---

## 1. 사전 준비

### 1.1 소프트웨어 환경

| 항목 | 버전 / 비고 |
|---|---|
| OS | Ubuntu 22.04 |
| ROS 2 | Humble |
| Docker | 20.10 이상 (postgres 컨테이너용) |
| Python | 3.10+ |
| ffmpeg / ffprobe | 시스템 PATH 에 설치 |
| 디스크 | 테스트 bag 1 개당 bag 크기의 2~3 배 여유 |

### 1.2 후처리기 빌드

```bash
cd ~/development/ros/rdfp_ws
sudo apt install python3-yaml python3-opencv ffmpeg
pip install --user 'mcap' 'mcap-ros2-support' 'pydantic>=2' 'psycopg[binary]>=3'
colcon build --packages-select rdfp
source install/setup.bash
```

### 1.3 DB 컨테이너 기동

```bash
docker run -d --name rdfp_pg \
  -e POSTGRES_USER=rdfp \
  -e POSTGRES_PASSWORD=rdfp \
  -e POSTGRES_DB=rdfp \
  -p 5432:5432 \
  postgres:16

export RDFP_DB_DSN="postgresql://rdfp:rdfp@localhost:5432/rdfp"

# 스키마 적용 (빈 DB 초기화) — 두 방법 중 택일
ros2 run rdfp dataset init-db --dsn-env RDFP_DB_DSN
#   또는
psql "${RDFP_DB_DSN}" -f src/rdfp/rdfp/postproc/sql/schema.sql

# 스키마 확인
psql "${RDFP_DB_DSN}" -c '\dt'
```

### 1.4 테스트 bag 선정 기준

| 구분 | 조건 |
|---|---|
| 스모크 테스트용 (짧음) | 30 분 이하, 에피소드 2~5 개, `/session` + 상태 토픽 + 가능하면 카메라 1 개 포함 |
| 성능 측정용 (긺) | 수 시간 ~ 1 일, 에피소드 20 개 이상 |
| QoS 오버라이드 | `/session` 이 TRANSIENT_LOCAL 로 기록되어 있어야 (운영 방안 5.5 참조) |

선정된 bag 의 경로를 환경변수로 고정해둔다.

```bash
export SMOKE_BAG_ROOT=/data/rosbag         # 스모크용 rosbag 루트
export PERF_BAG_ROOT=/data/rosbag_long     # 성능 측정용 rosbag 루트
```

## 2. PostgreSQL 통합 검증

### 2.1 최소 스모크

```bash
# 짧은 범위 설정 (rosbag / dataset 샘플을 합쳐 쓰는 통합 스모크 파일)
cp docs/rosbag2/dataset_config.sample.yaml /tmp/smoke.yaml
# /tmp/smoke.yaml 에서 rosbag_dir / output_mp4_dir / db 섹션을 실 환경에 맞춰 편집

# list-episodes 로 경계 확인
ros2 run rdfp rosbag list-episodes --config /tmp/smoke.yaml
# → IDX / START_NS / DUR_SEC / TASK_LABEL 테이블이 출력되어야 한다.

# 적재
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# → 실행 요약 JSON 이 stdout 으로 출력된다.

# DB 통계 확인
ros2 run rdfp dataset stats --config /tmp/smoke.yaml
```

### 2.2 스키마 사전 검증 테스트

DB 에 필요한 테이블 일부를 의도적으로 누락시킨 뒤 실행해 **즉시 실패** 하는지 확인한다.

```bash
psql "${RDFP_DB_DSN}" -c 'DROP TABLE pose_stampeds CASCADE;'
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# → PostProcError: required tables not found / missing columns ...
# 복구
ros2 run rdfp dataset init-db --dsn-env RDFP_DB_DSN
```

### 2.3 `on_existing_episode` 정책 검증

#### 2.3.1 `skip` (기본값)

```bash
# 1차 적재
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# 2차 적재
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# → 요약에서 skipped 가 1차 episodes 와 동일, inserted 행 수 증가 없음
```

#### 2.3.2 `replace`

```bash
# /tmp/smoke.yaml 의 on_existing_episode 를 replace 로 편집 후
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# → 요약에서 replaced 가 1 이상, inserted 는 1차 때와 동일한 수치가 재등장
# DB 에서도 행 수가 같아야 한다 (cascade 삭제 후 재삽입).

# mp4 디렉터리도 새 mtime 으로 재생성되었는지 확인
stat /data/postproc/videos/episode_00000001/*.mp4 | grep Modify
```

#### 2.3.3 `error`

```bash
# error 로 편집 후 기존 DB 에서 재실행
ros2 run rdfp dataset import --config /tmp/smoke.yaml
# → exit code 1, 'episode already present' 메시지
```

### 2.4 FK CASCADE 실제 검증

```bash
psql "${RDFP_DB_DSN}" <<'SQL'
SELECT id FROM sessions ORDER BY id DESC LIMIT 1;
-- 위 id 로 자식 행이 있는지 확인
SELECT COUNT(*) FROM pose_stampeds WHERE episode_id = (SELECT MAX(id) FROM sessions);
-- sessions 삭제 → cascade 되는지 확인
DELETE FROM sessions WHERE id = (SELECT MAX(id) FROM sessions);
SELECT COUNT(*) FROM pose_stampeds WHERE episode_id = (SELECT COALESCE(MAX(id), 0) FROM sessions);
SQL
```

## 3. 실 rosbag 스모크 테스트

### 3.1 검증 포인트

| 항목 | 기대 |
|---|---|
| `list-episodes` 결과 | 수동으로 bag 을 `ros2 bag play` 해봤을 때 관찰되는 IN_EPISODE 구간과 일치 |
| 에피소드 수 | summary `episodes` 와 `list-episodes` 결과 수가 일치 |
| DB 행 수 | 토픽 주파수 × 에피소드 길이 ± 수 % |
| mp4 duration | `ffprobe` 기준 duration 이 `stop_ts - start_ts` 에 근접 (CFR 선언이라 약간 오차 허용) |
| `metadata.json` + `frames.jsonl` | `metadata.nominal_frame_count == frames.jsonl 라인 수`, 최초/최종 stamp 가 에피소드 구간 내 |

### 3.2 런북

```bash
# 1) 에피소드 목록 확인
ros2 run rdfp rosbag list-episodes --config /tmp/smoke.yaml
# → 예상된 에피소드가 모두 감지되는지 확인.

# 2) 적재
ros2 run rdfp dataset import --config /tmp/smoke.yaml | tee /tmp/run_summary.json

# 3) DB 행 수 검증
psql "${RDFP_DB_DSN}" <<'SQL'
SELECT 'sessions'           AS t, COUNT(*) FROM sessions
UNION ALL SELECT 'pose_stampeds',      COUNT(*) FROM pose_stampeds
UNION ALL SELECT 'twist_stampeds',     COUNT(*) FROM twist_stampeds
UNION ALL SELECT 'joint_states',       COUNT(*) FROM joint_states
UNION ALL SELECT 'joint_trajectories', COUNT(*) FROM joint_trajectories
UNION ALL SELECT 'gripper_cmds',       COUNT(*) FROM gripper_cmds
UNION ALL SELECT 'gripper_states',     COUNT(*) FROM gripper_states;
SQL

# 4) mp4 / metadata / frames 검증 (에피소드 id 는 실제 값으로 치환)
EP=1
EP_DIR="/data/postproc/videos/episode_$(printf '%08d' $EP)"
ls -la "$EP_DIR"
for mp4 in "$EP_DIR"/*.mp4; do
    echo "== $mp4 =="
    ffprobe -v error -select_streams v:0 -count_frames \
        -show_entries stream=nb_read_frames,width,height,duration,r_frame_rate \
        -of default=noprint_wrappers=1 "$mp4"
done
for meta in "$EP_DIR"/*.metadata.json; do
    base="${meta%.metadata.json}"
    frames="${base}.frames.jsonl"
    python3 - <<PY
import json
with open("$meta") as f:
    m = json.load(f)
with open("$frames") as f:
    rows = [json.loads(line) for line in f if line.strip()]
print("$meta", "nominal:", m['nominal_frame_count'], "frames.jsonl 라인 수:", len(rows),
      "res:", m['width'], m['height'], "frame_id:", m.get('frame_id', ''))
print("  first stamp:", rows[0]['stamp'] if rows else None)
print("  last  stamp:", rows[-1]['stamp'] if rows else None)
PY
done

# 5) JSONL 실행 로그 확인
tail -n 20 /data/postproc/videos/_logs/postproc_run.jsonl
```

### 3.3 결과 기록 템플릿

`/tmp/run_summary.json` 과 `ffprobe` 출력을 보존해 놓으면 회귀 테스트의 gold 샘플로 쓸 수 있다.

```
docs/rosbag2/validation/
├── smoke_<날짜>/
│   ├── dataset_config.yaml          (사용한 설정)
│   ├── summary.json                 (import 의 stdout)
│   ├── row_counts.txt               (psql 결과)
│   ├── ffprobe.txt                  (각 mp4 의 ffprobe 출력)
│   └── run.jsonl                    (JSONL 로그 복사)
```

## 4. 병렬 처리 성능 측정

Phase 3 수락 기준:
- 단일 프로세스 대비 N 배 가속 (N 은 에피소드 수·코어 수에 의존)
- 메모리 상한 초과 없이 완주
- 결과 일관성: `parallelism=1` 과 `>1` 의 DB/mp4 결과가 동일

### 4.1 벤치마크 환경 확인

```bash
nproc                              # CPU 코어 수
free -h                            # 사용 가능 메모리
df -h /data                        # 디스크 여유
```

### 4.2 측정 프로토콜

같은 `PERF_BAG_ROOT` 와 시간 범위로 `parallelism` 만 바꿔가며 반복 실행한다.

```bash
cp docs/rosbag2/dataset_config.sample.yaml /tmp/perf.yaml
# /tmp/perf.yaml 의 rosbag_dir, on_existing_episode=replace 로 설정

for N in 1 2 4 8; do
    # parallelism 값 변경
    python3 -c "
import yaml, pathlib
p = pathlib.Path('/tmp/perf.yaml')
c = yaml.safe_load(p.read_text()); c['parallelism'] = $N
p.write_text(yaml.safe_dump(c, sort_keys=False))
"
    echo "=== parallelism=$N ==="
    /usr/bin/time -v ros2 run rdfp dataset import --config /tmp/perf.yaml \
        2> /tmp/perf_$N.stderr \
        | tee /tmp/perf_$N.stdout
    echo "completed=$(date +%s)"
done
```

`/usr/bin/time -v` 출력에서 다음 지표를 기록한다.

| 지표 | time 출력 필드 |
|---|---|
| Wall clock | `Elapsed (wall clock) time` |
| CPU 사용률 | `Percent of CPU this job got` |
| Peak RSS | `Maximum resident set size` |

### 4.3 결과 일관성 검증

각 parallelism 수치에서 summary 의 `episodes`, `inserted`, `mp4_files` 가 **모두 동일** 해야 한다.

```bash
for N in 1 2 4 8; do
    jq '{episodes, inserted, mp4_files}' /tmp/perf_$N.stdout
done | sort -u | wc -l
# → 1 (완전 동일) 이 정답
```

### 4.4 실시간 메모리 프로파일 (선택)

`psutil` 기반 간단 샘플러를 병렬로 돌려 peak RSS 외에 시계열 그래프를 얻는다.

```bash
python3 - <<'PY' &
import os, psutil, time, sys
pid = int(os.environ.get('TARGET_PID', os.getppid()))
while psutil.pid_exists(pid):
    p = psutil.Process(pid)
    total = p.memory_info().rss + sum(c.memory_info().rss for c in p.children(recursive=True))
    print(time.time(), total, flush=True)
    time.sleep(1.0)
PY
SAMPLER_PID=$!

ros2 run rdfp dataset import --config /tmp/perf.yaml
kill $SAMPLER_PID
```

## 5. SQLite 인덱스 연동 검증

### 5.1 인덱스 구축

운영 방안 10.5 규격에 맞는 SQLite 인덱스가 있어야 한다. 스크립트 예시:

```bash
# 실제 구축 스크립트 (docs/rosbag2/uncheckeds/build_bag_catalog.sh 참조)
export CATALOG=/data/rosbag/_index.sqlite
bash docs/rosbag2/uncheckeds/build_bag_catalog.sh "${SMOKE_BAG_ROOT}" "${CATALOG}"
```

### 5.2 비교 측정

```bash
# (A) 인덱스 미사용 (metadata.yaml fallback)
#   /tmp/smoke.yaml 에서 catalog_index_path: null
time ros2 run rdfp rosbag list-episodes --config /tmp/smoke.yaml

# (B) 인덱스 사용
#   catalog_index_path: /data/rosbag/_index.sqlite
time ros2 run rdfp rosbag list-episodes --config /tmp/smoke.yaml
```

에피소드 목록은 동일해야 하고, (B) 가 (A) 보다 빨라야 한다 (긴 rosbag 에서 유의미).

```bash
# 인덱스 경로를 존재하지 않는 파일로 지정해 fallback 동작 확인
```

## 6. 품질 게이트 현장 검증

### 6.1 stamp 역행 감지

실 bag 중 stamp 역행이 의심되는 구간이 있거나, 인위적 테스트가 필요하면 `idle_gap_sec` 를 낮게 설정하여 경고가 기록되는지 확인한다.

```yaml
# /tmp/smoke.yaml
quality_gate:
  stamp_regression: true
  idle_gap_sec: 0.5           # 저주파 토픽에서는 쉽게 트리거
```

```bash
ros2 run rdfp dataset import --config /tmp/smoke.yaml
grep '"event":"quality_warning"' /data/postproc/videos/_logs/postproc_run.jsonl | head
```

경고 레코드의 `kind`, `topic`, `stamp_ns`, `prev_stamp_ns` 값이 타당한지 확인한다.

### 6.2 summary 의 warnings 카운터

```bash
jq '.warnings' /tmp/smoke.stdout
```

## 8. 결과 기록과 공유

각 검증 회차의 산출물을 `docs/rosbag2/validation/<회차명>/` 하위에 보관한다.

권장 포함 파일:

- 사용한 설정 (`rosbag_config.yaml` / `dataset_config.yaml`)
- summary 원본 (`summary.json`)
- DB 행 수 (`row_counts.txt`)
- `ffprobe` 출력 모음 (`ffprobe.txt`)
- JSONL 로그 (`run.jsonl`)
- `time -v` 출력 (`time_<N>.stderr`)
- 환경 정보 (`env.txt` — `uname -a`, `nproc`, `free -h`, `lsb_release -a`)

이 산출물이 축적되면 **회귀 테스트 gold** 로 승격해 Phase 1~3 기능의 장기 안정성을 감시할 수 있다.

## 9. 운영 이관 점검

위 모든 검증이 통과하면 아래 항목을 최종 점검하고 운영 이관한다.

- [ ] 설계서·본 문서의 기술 선택이 실제 구현과 일치한다.
- [ ] `rosbag_config.sample.yaml` / `dataset_config.sample.yaml` 이 실 운영 환경에 맞춰 편집된 버전 컨트롤 대상 파일로 분리되어 있다.
- [ ] `RDFP_DB_DSN` 이 systemd/환경변수 관리 시스템에 등록되어 있다.
- [ ] 인덱스 빌드 스크립트가 cron/systemd-timer 로 주기 실행되도록 등록되어 있다 (장시간 운영 시).
- [ ] JSONL 로그 디렉터리의 retention 정책 (예: 90 일) 이 정의되어 있다.
